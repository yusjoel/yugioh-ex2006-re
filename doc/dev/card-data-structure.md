# 卡牌数据结构调查报告

本文记录对《游戏王 EX2006》（2343.gba）中卡牌相关数据的调查结果，
涵盖卡牌属性数据表、卡名字符串表、卡牌图像格式，以及 LZ77 压缩图形区域。

---

## 一、卡牌属性数据表

### 位置与规模

| 项目 | 值 |
|------|-----|
| ROM 偏移起始 | `0x018169B6` |
| ROM 偏移结束 | `0x01832602` |
| 记录大小 | 22 字节（11 × uint16 LE） |
| 总记录数 | 5170 条 |
| 唯一槽位数 | 2054 个（copy_index=0 的主记录） |
| 槽位 ID 范围 | `0x0000` – `0x19FE` |

### 字段布局

每条记录 22 字节，以 **uint16 小端**格式排列：

| 字节偏移 | 字段名 | 说明 |
|---------|--------|------|
| +00 | `zero_0` | 恒为 `0x0000` |
| +02 | `slot_id` | 卡槽编号（与 `data.md` 中 Slot 列一致） |
| +04 | `copy_index` | 副本索引（0=主记录，1/2/3=第几个副本） |
| +06 | `one` | 恒为 `0x0001`（可能是"有效"标记） |
| +08 | `atk` | 攻击力；Spell/Trap 卡为 `0xFFFF` |
| +10 | `def` | 守备力；Spell/Trap 卡为 `0xFFFF` |
| +12 | `level` | 星数（怪兽卡） |
| +14 | `attribute` | 属性代码（见下表） |
| +16 | `race` | 种族代码（见下表） |
| +18 | `unknown` | 大部分为 0，Spell/Trap 有非零值 |
| +20 | `zero_1` | 恒为 `0x0000` |

### 属性编码（部分确认）

| 值 | 属性 | 验证来源 |
|----|------|---------|
| 1 | LIGHT（光） | Blue-Eyes White Dragon |
| 3 | DARK（暗） | Kuriboh |

> ⚠️ 完整属性/种族编码表尚未全部验证，需对照 `data.md` 进一步比对。

### 种族编码（部分确认）

| 值 | 种族 | 验证来源 |
|----|------|---------|

## 二、卡名字符串表（多语言）

### 2.1 欧洲语言卡名（CP1252 编码）

| 项目 | 值 |
|------|-----|
| ROM 偏移起始 | `0x015BB5AC` |
| 字符串编码 | CP1252（兼容 Latin-1，含德语 `ä ö ü` 等扩展字符） |
| 排列顺序 | 与卡牌数据表槽位顺序一致 |

每张卡依次排列 **5 个语言**的 null 终止字符串：

| 顺序 | 语言 |
|------|------|
| 1 | EN（英语） |
| 2 | DE（德语） |
| 3 | FR（法语） |
| 4 | IT（意大利语） |
| 5 | ES（西班牙语） |

**对齐规则**：每个字符串（含终止 `\0`）若字节数为奇数，则补一个额外 `\0`，保持 2 字节对齐。

#### 已验证示例

```
ROM 0x015BB5AC  → "Blue-Eyes White Dragon\0"     (EN, 24字节，已对齐)
                → "Blauäugiger Weißer Drache\0\0" (DE, 偶数对齐)
                → "Dragon Blanc aux Yeux Bleus\0\0" (FR)
                → "Drago Bianco Occhi Blu\0\0"    (IT)
                → "Dragón Blanco de Ojos Azules\0" (ES)

下一张卡（slot 0x0FA8 = Mystical Elf）紧接其后
```

### 2.2 日语卡名（自定义双字节编码）

游戏还包含 **日语版**卡名字符串，存储于两个连续区域：

| 项目 | 值 |
|------|-----|
| 索引表起始 | `0x0183885C` |
| 索引表条目数 | 2098 条（2098 × 4 字节 = 8392 字节） |
| 索引表结束 | `0x0183A924` |
| 字符串池起始 | `0x0183A924` |
| 字符串池大小 | ~43 KB（最大偏移量 = 43010）|
| 字符串编码 | 自定义双字节编码，范围 `0xF081`–`0xF2A1` |
| 终止符 | `0x00`（单字节） |

**索引表格式**：每条 4 字节（uint32 LE），为相对于字符串池基址的字节偏移量。

**字符串解码**：每个字符 2 字节（高字节在前），值范围 0xF081–0xF2A1，对应日语假名/汉字。

**访问算法**（函数 ROM `0x000EEB3C`）：
```
card_name_ptr = string_pool_base + offsets[name_index]
其中 string_pool_base = 0x0183A924，offsets = uint32 数组（0x0183885C 起）
```

**唯一字符串数**：2054（有 44 对条目共享同一偏移，对应同名异画卡）。

## 三、卡牌图像数据

### 图像区位置与格式

| 项目 | 值 |
|------|-----|
| ROM 偏移起始 | `0x01000000` |
| ROM 偏移结束 | `0x01463480` |
| 总大小 | ~4.4 MB（= 2054 × 2240 字节） |
| 格式 | **8bpp GBA tiles**（256 色调色板） |
| 每卡尺寸 | 5 tiles × 7 tiles = **40×56 像素** |
| 每卡字节数 | 5 × 7 × 64 = **2240 字节**（stride） |
| 卡牌总数 | **2054** 张（与唯一槽位数一致） |

图像区结束后紧接全零填充，边界精确匹配（`2054 × 2240 = 0x463480`）。

### 8bpp Tile 格式说明

8bpp tiles：每个像素 1 字节，值为调色板索引（0–255）。

每个 8×8 tile 占 **64 字节**，行优先排列：
```
字节 0  → 第0行第0列（色彩索引）
字节 1  → 第0行第1列
...
字节 63 → 第7行第7列
```

tile 按列优先（Column-major）铺排成整张卡：
```
tile 0  = 左上角 (0,0)–(7,7)
tile 1  = (8,0)–(15,7)
...
tile 4  = (32,0)–(39,7)    ← 第一行最右 tile
tile 5  = (0,8)–(7,15)
...
```

### 调色板（未解决）

尚未确认每张卡调色板的存储位置。

已探索区域：

| 区域 | 情况 | 结论 |
|------|------|------|
| `0x00FF0000 – 0x01000000` | 共 128 个 512 字节块 | 所有 pal[0] ≠ 0，不符合 GBA 调色板惯例 |
| `0x01463480` 之后 | 全零至 `0x014634A0` | 非调色板 |
| stride 内嵌（2240+512 = 2752 B/卡） | 未充分验证 | 待测试 |

> **待办**：验证每卡是否内嵌 512 字节调色板（stride = 2752），
> 或调色板存储在其他区域（如 `0x01463480` 之后的低熵区）。

### 图像索引映射

验证表明图像按**顺序索引**排列（不是 slot_id 直接作为索引）：

```
image_index 0 → 卡名表第0张卡
image_index 1 → 卡名表第1张卡
...
地址公式: ROM 偏移 = 0x01000000 + image_index × 2240
```

以 slot_id 直接作索引（Blue-Eyes = 0x0FA7 = 4007）会超出图像区，故排除。

### ⚠️ 当前导出状态（未完成）

**本区数据（8bpp，2240 B/卡）推测为卡牌列表小图或非主图用途**，调色板尚未确认，
目前导出均无法正确着色。灰阶渲染可见怪兽轮廓和卡牌边框，证实了格式与布局，
但此图像区起始处（index 0–若干）实际存储的是 UI 字体/数字图形（蓝色背景上的 0–9），
非卡牌插画，实际插画起始 index 未确认。

### 3.2 另一类大卡图数据（详情页用）

> **独立于本节主体描述的 0x01000000 区**。Phase B2 已验证（`p1-phase-b2-findings.md`）。

卡牌详情页显示的大卡图是**另一批数据**，不在 `0x01000000` 区：

| 项目 | 值 |
|------|-----|
| ROM 地址基址 | `0x08510640`（GBA 指针） |
| ROM 文件偏移 | `0x00510640` |
| 单卡尺寸 | **80×80 像素**（10×10 tiles） |
| 单卡字节数 | **4800 字节**（stride） |
| 格式 | **6bpp 自写压缩**（非 BIOS SWI 解压；每 6 ROM bytes → 8 像素） |
| 加载函数 | `FUN_0801d290`（`asm/all.s` 行 15429） |
| 索引表 | `0x095B5C00`（每项 u16 LE，`card_id × 2 + flag` 作字节偏移） |
| 调色板 | `0x084C76C0`（256 色 BG 调色板） |
| card_id 来源 | EWRAM 卡片对象 `(word0 << 15) >> 18` |
| 导出脚本 | `tools/ad-hoc/decode_card_6bpp.py`（单卡验证） |

6bpp 解码公式与实现细节见 `p1-phase-b2-findings.md` §3.4–3.5。

## 四、LZ77 压缩图形区域

### 主要压缩区（游戏背景画面）

| 项目 | 值 |
|------|-----|
| ROM 偏移范围 | `0x1B8FB8C` – `0x1DBF019` |
| 区域大小 | ~2.3 MB |
| 内容 | 游戏背景画面（推测） |
| 压缩格式 | GBA LZ77（BIOS 解压，header byte `0x10`） |

该区域直接跟在对手图形（`0x1B8FB8B` 结束）之后，在卡组名字符串表（`0x1DBF01A`）之前。

**GBA LZ77 格式说明**：
```
字节 0        = 0x10（LZ77 类型标识）
字节 1–3（LE）= 解压后大小（3字节，最大 16MB）
字节 4+      = 压缩数据（每8位一个 flag byte，0=字面量，1=反向引用）
```

### 标题画面 LZ77

| 项目 | 值 |
|------|-----|
| ROM 偏移 | `0x1EC33DC` |
| 内容 | 标题画面压缩图形 |
| 数据来源 | `doc/um06-romhacking-resource/title-screen.md` |

> 标题画面需专门工具解压处理，已列入 PLAN.md T5。

## 五、密码系统调查

### 调查概述

游戏拥有完整的密码机系统（代码位于 ROM `0x000DBA64`），允许玩家输入 8 位数字密码获取卡牌。
相关 UI 字符串已在 ROM `0x01DCB29A` 处确认：

- `"Enter the password."` / `"DEL"` / `"OK"`
- `"The card password is incorrect or cannot be used in this game."`
- `"Password Lists"` / `"Password List 1"` / `"List - August, 199x"`

### 关键代码结构

| 功能 | ROM 地址 | 说明 |
|------|---------|------|
| 密码机 UI 初始化 | `0x000DBA64` | 密码机界面 |
| 密码机状态机 | `0x000DE2C0` | 输入处理 |
| 数值累加（+digit） | `0x000F8F1C` | 将参数加到当前数值，上限 99999999 |
| 数值减少（-digit） | `0x000F8F50` | 将参数从当前数值减去，下限 0 |
| 直接设值 | `0x000F8F78` | 直接设置当前数值 |
| 数字显示更新 | `0x000F9C68` | 上述三函数均调用此函数 |
| 名称查找 | `0x000EEB3C` | 按索引查找日语卡名字符串 |

> **注**：EWRAM `0x02006C38` 存储的是游戏数值（可能为当前 DP/金钱），而非玩家输入的密码数值。

### 密码表搜索结论

以下格式均已全面搜索 ROM，均**未找到** TCG 标准 8 位密码（如 Blue-Eyes = `89631139`）：

| 搜索格式 | 结果 |
|---------|------|
| LE uint32 (`A3 A9 57 05`) | 未找到 |
| BE uint32 (`05 57 A9 A3`) | 未找到 |
| BCD 打包 (`89 63 11 39`) | 未找到 |
| ASCII 字符串 (`"89631139"`) | 未找到 |
| XOR 变换 | 未找到 |

### 可能的原因

1. **密码算法化**：密码由卡牌 ID 通过某种数学变换生成（逆向查找），无需存表。
   函数 ROM `0x000EF370` 中发现可疑常量 `0x21A29A69`（可能为乘法逆元）。
2. **独立密码系统**：游戏可能使用与 TCG 不同的简化密码（如"Password Lists"机制）。
3. **数据压缩**：密码表可能以 LZ77 压缩存储，但全 ROM 扫描未找到合理大小的 LZ77 块。
